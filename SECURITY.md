## SLSA L2 build provenance

Every release ships a SLSA Level 2 build-provenance attestation
alongside the signed source tarball. The attestation is
generated by this repository's
[release workflow](./.github/workflows/release.yml) on
GitHub-hosted runners, signed via Sigstore's keyless OIDC
flow, and published to the public Rekor transparency log.

The provenance attestation certifies, in machine-checkable
form, that the published tarball was produced by an
unmodified CI run triggered by the signed `v*` tag on this
repository. It is a stronger claim than the GPG signature
alone: even with the publisher's GPG key compromised, moving
a tag still requires landing a new commit and re-running CI
under the attacker's OIDC identity, which would be visible in
the transparency log.

### Verify provenance manually

```bash
# Download the signed tarball from the release page.
gh release download v0.1.0 \
    --repo nelsonduarte/capa_agent_demo \
    --pattern '*.tar.gz'

# Verify the attestation against the public Sigstore log.
gh attestation verify capa_agent_demo-v0.1.0.tar.gz \
    --owner nelsonduarte
```

The verification confirms three claims:

1. The tarball SHA-256 matches what was attested.
2. The attestation was issued by GitHub Actions running in
   the `nelsonduarte/capa_agent_demo` repository.
3. The trigger event was a tag push.

## The three-layer supply-chain stack

This demo's `capa.toml` declares `verify_key` on both seed
libraries it consumes (`capa_http` and `capa_log`); `capa
install` then enforces all three layers on every install:

1. **Lockfile SHA enforcement** — refuses a moved upstream tag.
2. **GPG `verify-tag`** — refuses an unsigned or wrong-key tag.
3. **SLSA L2 `attestation verify`** — refuses a tampered or
   wrong-owner Sigstore attestation.

The same three-layer stack applies to *this* repo when a
downstream consumer pins `capa_agent_demo` as a dep in their
own `capa.toml`.
